## req-wireframe/SKILL.md 审查报告（无 skill 基线版）

### 总体评价

整体结构清晰，五步流程逻辑合理，CSS 变量、组件选用原则、文件命名规则等关键约束定义细致，可操作性强。存在 2 个严重问题、3 个中等问题、4 个轻微问题。

---

### 严重问题

**P1：截图工具未指定（"浏览器工具"表述模糊）**
步骤 3d 要求"用浏览器工具打开该 URL"并截图，但未指明具体工具或 API。Claude Code 环境中截图依赖特定 MCP 工具（如 `mcp__Claude_in_Chrome` 或 `mcp__computer-use__screenshot`），不同部署环境不同。当前写法可能导致 LLM 无法确定该调用哪个工具，截图步骤完全无法执行。
建议：明确指定工具名，或说明"使用当前可用的浏览器截图工具"并补充无截图能力时的降级方案。

**P2：`output/` 目录的相对路径基准未统一说明**
步骤一的路径说明只提到 `shared/` 以 `req-skill/` 为基准，但后续多处使用的 `output/wireframes/` 没有同等声明。父级 `req-skill/SKILL.md` 将 `output/` 定义为"项目根目录"，与子 skill 的用法存在潜在冲突。
建议：在步骤一路径说明中补充 `output/` 的基准，或与父 skill 对齐明确说明"output/ 即项目根目录下的 output/"。

---

### 中等问题

**P3：占位符前无加粗行时的屏幕名称提取逻辑失效**
步骤二要求"往上找最近的 `**...**` 加粗行"，但若某需求卡片只有一个无 Tab 的页面、图前没有状态标题，则提取逻辑失效，可能产生空文件名或自行编造名称。
建议：补充降级规则，如"若找不到加粗行，以需求卡片名称加序号作为屏幕名称"。

**P4：用户确认交互节奏混用**
步骤二结尾要求等待用户批量确认后才开始；步骤 3d 结尾又要求每张截图后询问"是否继续下一个"。批量确认和逐个确认混用，10+ 个占位符时极为繁琐。
建议：明确默认节奏，如"确认清单后自动批量执行，失败时暂停，逐个确认仅在用户明确要求时启用"。

**P5：HTML 渲染质量未验证**
步骤 3c 只生成 HTML 文件，步骤 3d 才截图。当前质量检查只验证"PNG 存在且大小 > 0"，无法发现页面渲染错误（空白页、标签未闭合等）。
建议：在质量自检清单中补充"截图中应可见页面标题和至少一行数据内容（非空白页）"。

---

### 轻微问题

**P6：拼音命名规则不确定，示例本身有拼写错误**
"折让"拼为"zhekang"（应为"zherang"），说明 LLM 自由生成拼音缩写时容易出错，且不同运行间命名不一致，可能导致 PRD 中图片路径断链。
建议：改用系统需求编号或固定的全拼规则以保证确定性。

**P7：用原始文档时 fallback 路径未传递给后续步骤**
步骤一说如果 `prd-updated.md` 不存在，询问用户是否直接用原始文档并要求提供路径，但步骤四的替换逻辑仍以固定路径为前提，未说明如何使用用户提供的自定义路径。
建议：补充"将用户提供的路径记为输入文档路径，后续操作均使用此路径"。

**P8：版本号递增规则未覆盖原始文档**
步骤四只说"修改 `prd-updated.md` 时递增版本号"，未说明修改原始文档时是否也需要处理版本号。
建议：补充"若修改原始文档，同样检查并递增版本号字段，若无则跳过"。

**P9：`shared/wireframe-style.html` 的使用方式不明确**
未说明是"完整复制 `<style>` 块"还是"参考风格自行编写"，前者保证一致性，后者可能导致样式偏差。
建议：明确说明"将 `shared/wireframe-style.html` 中的 `<style>` 块完整复制到生成的 HTML 中，再追加页面特有样式"。

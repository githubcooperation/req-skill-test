## 一、总体评价

req-skill 是一套面向产品经理的需求文档全流程质量工具，覆盖"审查 → 补全 → 原型 → 测试用例 → 流程文档"五个阶段，设计思路清晰，整体质量较高。主入口的意图路由机制和状态感知机制均有完整定义，各子 Skill 的执行流程规范，共享资源的接口契约基本完整。

主要优势：
- 工具链设计完整，五个阶段形成闭环，且允许任意子集单独触发
- 共享资源（schema、template）与执行文件分离，便于单独维护
- 格式约束明确（ISS-ID、TC-ID 格式规范、强制必填字段清单），减少歧义
- 提供了完整的示例原始材料（raw-materials），便于测试和演示

主要问题（按严重度排列）：
1. **P1**：`req-wireframe` 截图机制依赖外部浏览器工具，但工具可用性未作保障性处理
2. **P1**：`req-complete` 自动补全模式对安全相关项的参考路径描述存在条件歧义
3. **P1**：`shared/prd-template.md` 与 `req-flowdoc` 对第 4 章结构的假设存在轻微耦合风险
4. **P2**：`reference/business-rules.md` 完全为模板占位内容，未对 req-review 的行为产生实际约束
5. **P2**：`req-testcase` 汇总表格式与 `shared/testcase-template.md` 的模板格式存在字段不一致
6. **P2**：部分子 Skill 的路径说明（"路径说明"注释块）内容重复，维护成本较高
7. **P2**：`shared/raw-materials/` 目录的用途在主 SKILL.md 中没有任何说明

## 六、req-wireframe 审查

**文件路径**：`req-wireframe/SKILL.md`、`shared/wireframe-style.html`

### 6.1 执行流程规范性

**评价：良好，存在一处关键风险**

HTML 生成规范非常详细，CSS 变量统一定义，组件选用原则和文件命名规则清晰。截图后验证（检查 PNG 是否存在且非空）的机制值得肯定。

**发现问题**：

| # | 严重度 | 问题描述 |
|---|--------|---------|
| W-01 | P1 | 第 3d 步截图机制依赖"浏览器工具"（通过 `file://` URL 打开 HTML 并截图），但该工具在不同环境下可能不可用（无头服务器、CI 环境、无 GUI 的终端）。SKILL.md 的处理是"两次均失败则告知用户手动截图"，但"两次"的失败判断条件（PNG 不存在或大小为 0）在某些环境下可能永远失败，导致所有线框图都需要手动截图，使 req-wireframe 的价值大幅降低。缺少对工具可用性的前置检查和明确的降级方案说明。 |
| W-02 | P1 | 截图步骤说"用 Bash 执行 `realpath output/wireframes/{文件名}.html`"，但这个 `output/wireframes/` 路径是相对路径，相对于哪个目录？如果 shell 工作目录不是项目根目录，`realpath` 会返回错误路径。SKILL.md 自己也提到"不要使用 `pwd` 拼接路径，因为 shell 工作目录不一定是项目根目录"，但紧接着的 `realpath` 命令同样依赖当前工作目录来解析相对路径，前后矛盾。 |
| W-03 | P2 | 文件命名规则示例 `折让管理 > 折让核算明细 > [全部]状态` → `zhekang-hesuan-quanbu.html`，但"折让"的拼音应为 `zherang`，示例中写成了 `zhekang`。这是一个错别字，可能导致 AI 模仿错误拼音命名。 |
| W-04 | P2 | 第四步"更新 PRD 文档"要求将 `{低保真原型图}` 替换为 `![{屏幕名称}线框图](output/wireframes/{文件名}.png)`，但图片路径 `output/wireframes/...` 是相对路径，相对于 PRD 文件的位置。如果 PRD 文件不在项目根目录，这个相对路径将失效。应使用绝对路径或说明相对路径的基准。 |
| W-05 | P2 | 第二步扫描占位符时，"每个占位符的定位方式"要求"往上找最近的 `**...**` 加粗行"作为屏幕名称，但如果需求卡片中紧接在 `{低保真原型图}` 之前没有加粗行（例如只有一个默认状态没有 Tab），AI 的回退行为未定义。 |

CSS 变量设计规范，12 个参考组件覆盖了常见 B 端系统的 UI 构件。弹窗和空状态组件通过 HTML 注释形式保存（不渲染但可复制）是聪明的设计，避免样式文件本身变成一个复杂的多态页面。

**发现问题**：

| # | 严重度 | 问题描述 |
|---|--------|---------|

## 七、req-flowdoc 审查

**文件路径**：`req-flowdoc/SKILL.md`、`shared/flow-doc-template.md`

### 7.1 执行流程规范性

**评价：优秀**

三步识别流程（优先扫描位置→识别规则→信息完整度评估）设计严谨，"展示识别结果等待用户确认"的机制防止了误识别导致的批量无效输出。字段填写规则的"推断规则"列（金额→Decimal、日期→Date 等）是特别实用的设计。

**发现问题**：

| # | 严重度 | 问题描述 |
|---|--------|---------|
| F-01 | P1 | SKILL.md 第四步要求"**严格**按 `shared/flow-doc-template.md` 的结构输出"，但 flow-doc-template 中的出参固定字段表包含 `timestamp` 字段（类型为 Long，最大长度为"—"），而 SKILL.md 的"出参固定字段"表的 `timestamp` 行最大长度列也是"—"。两处一致，但 flow-doc-template 中 `timestamp` 行的"是否必填"列有内容但 SKILL.md 的对应说明表中未明确说明是否允许修改出参固定字段。如果某些接口确实没有 timestamp，应说明是否允许删除该行。 |
| F-02 | P2 | 文件命名规则示例"折让核算流程 → `zhekang-hesuan.md`"与 req-wireframe 的同一错误相同——"折让"拼音应为 `zherang`。建议统一修正为 `zherang-hesuan.md`。 |
| F-03 | P2 | 第五步汇总中 TBD 数量的统计方式有注释说明（"计算所有生成文件中值为 `TBD` 或以 `TBD（` 开头的表格单元格数量"），这个统计规则相当具体，但它被嵌套在输出模板代码块内，AI 可能将其理解为"输出内容"而非"执行指令"。建议将统计规则移至代码块外作为独立说明。 |
| F-04 | P2 | 对"已有流程文档的处理"提供了 A/B/C/D 四个选项，但缺少一个"E) 追加新增流程，不修改现有文件"的选项。在迭代开发场景中，PRD 更新后需要补充新流程文档而不覆盖旧的，当前选项 B"只生成缺少的"接近此需求但语义略有差异。 |

## 十、整体设计评价

### 10.1 优秀设计点

1. **路径说明注释块**：每个子 Skill 在引用跨目录文件时都有"路径说明"注释块，明确说明路径相对基准。这是解决多层目录中路径歧义的实用设计。
2. **ISS-ID 映射机制**：req-review 生成 ISS-ID，req-complete 消费并在 changelog 中追溯，形成完整的问题生命周期管理。
3. **"宁可多记不可漏记"原则**：req-review 明确说明不确定是否是问题时以 P2 级别记录，将判断权交给用户，符合工具辅助而非替代人类决策的定位。
4. **模板中的占位符设计**：prd-template.md 使用 `{低保真原型图}` 作为占位符，req-wireframe 通过扫描替换，形成了清晰的异步协作接口。
5. **CSS 变量全灰度方案**：wireframe-style.html 使用统一 CSS 变量并强制全灰度，从根本上保证了"低保真"的视觉一致性。

### 10.2 架构级风险

| # | 严重度 | 问题描述 |
|---|--------|---------|
| A-01 | P1 | `output/` 目录的路径基准全工具链中没有统一定义。各子 Skill 都使用 `output/` 相对路径，但这个路径相对于谁（shell 工作目录、skill 目录、用户指定的 PRD 文件所在目录）？在不同的使用场景下，这个隐式假设可能导致文件写入到错误位置。建议在主 SKILL.md 中增加一条"output 目录基准：相对于用户当前工作目录"的明确说明。 |
| A-02 | P2 | 工具链对"多轮迭代"的支持不完整：changelog.md 设计了"追加模式"，但 prd-updated.md 是覆盖写入。如果用户做了第三轮补全，第二轮的 prd-updated.md 将被覆盖且无法恢复。建议引入版本快照机制（如 `prd-updated-v2.md`）或在每次覆盖前提示用户。 |
| A-03 | P2 | 五个子 Skill 之间的数据流方向是单向线性的（review → complete → wireframe/testcase/flowdoc），但现实中用户可能在任意阶段返回修改，例如在生成测试用例后发现需求问题、修改 prd-updated.md 后需要重新生成受影响的测试用例。当前工具链没有"局部重新生成"的触发机制说明。 |

# 通用审查规则

以下规则适用于任何类型的需求文档，不依赖具体项目的业务知识。

## 一、完整性检查

### 1.1 功能描述完整性
- 每个功能点是否有明确的**输入**（用户操作或系统触发条件）
- 每个功能点是否有明确的**输出**（系统响应、界面变化、数据变更）
- 每个功能点是否说明了**谁**可以执行（角色/权限）

缺少任一项，标记为 `missing`，严重度视功能重要性定 P0 或 P1。

### 1.2 流程完整性
- 有开始必有结束：每个流程是否有明确的终止条件
- 有正向必有逆向：是否描述了取消、回退、撤销的处理方式
- 有成功必有失败：每个操作的失败场景是否有说明
- 分支是否覆盖完整：如果存在条件判断，所有分支（包括 else）是否都有说明

缺少逆向流程或失败处理，标记为 `incomplete_flow`。

### 1.3 状态流转完整性
- 如果涉及状态变化（如订单状态、审批状态），是否列出了所有可能的状态
- 每个状态之间的转换条件是否明确
- 是否存在"死状态"（只进不出的状态，且非最终状态）
- 并发场景下的状态冲突是否有说明

状态流转不完整标记为 `incomplete_flow`，并发冲突未说明标记为 `missing`。

## 二、明确性检查

### 2.1 模糊用词
以下词汇如果出现且没有附带具体定义，标记为 `ambiguous`：

**程度类**："合理的"、"适当的"、"较好的"、"足够的"、"一定的"
**时间类**："尽快"、"及时"、"稍后"、"不久"、"短时间内"
**范围类**："等等"、"之类的"、"诸如此类"、"相关的"
**数量类**："大量"、"少量"、"若干"、"一些"、"多个"

每处模糊用词独立记一条缺陷，严重度根据上下文判断：
- 影响核心逻辑的 → P1
- 仅影响体验描述的 → P2

### 2.2 可量化性
涉及数值的描述是否给出了具体数字：
- 超时时间：具体多少秒/分钟
- 长度限制：最小/最大多少字符
- 数量限制：最多/最少多少条
- 频率限制：多少次/分钟或秒
- 精度要求：保留几位小数

缺少具体数值标记为 `missing_constraint`。

### 2.3 可测试性
每条需求是否可以被转化为"操作步骤 + 预期结果"的形式。如果一条需求读完后，测试人员无法写出测试用例，标记为 `untestable`。

常见不可测试的表述：
- "用户体验要好"
- "系统性能要高"
- "界面要美观"
- "操作要流畅"

## 三、一致性检查

### 3.1 术语一致性
- 同一概念在文档中是否使用了不同名称（如"用户"和"客户"混用，"订单号"和"交易号"混用）
- 如果有术语表，文档正文是否与术语表一致

术语不一致标记为 `contradiction`，严重度 P1。

### 3.2 数据一致性
- 同一字段在不同章节的描述（类型、长度、格式、可选/必填）是否一致
- 枚举值在不同上下文中的取值范围是否一致

数据描述矛盾标记为 `contradiction`，严重度 P0（因为会直接导致开发实现不一致）。

### 3.3 逻辑一致性
- 不同章节描述的规则是否存在冲突
- 优先级或执行顺序是否有矛盾

逻辑冲突标记为 `contradiction`，严重度 P0。

## 四、边界与约束检查

### 4.1 边界条件
涉及输入的地方是否说明了边界情况：
- 空值/null 如何处理
- 最大值/最小值时的行为
- 特殊字符（空格、emoji、HTML标签等）的处理
- 超长输入的处理

缺少边界说明标记为 `missing_constraint`，严重度 P1。

### 4.2 并发与时序
- 多用户同时操作同一数据时的行为
- 操作超时后的处理
- 重复提交的处理
- 网络中断后的恢复策略

如果功能涉及多用户协作或关键数据修改，缺少并发说明标记为 `missing`，严重度 P1。

### 4.3 数据约束
- 数据格式要求（如日期格式、手机号格式、邮箱格式）
- 数据关联关系（外键、依赖关系）
- 数据保留策略（过期删除、归档规则）

缺少数据格式定义标记为 `missing_constraint`。

## 五、非功能性检查

以下维度如果文档完全未提及，以 P2 记录一条汇总缺陷（不需要逐条拆分）：

- **权限**：是否说明了各角色的操作权限
- **性能**：是否提及响应时间、吞吐量等性能指标
- **兼容性**：是否说明了支持的浏览器、设备、操作系统范围
- **安全性**：是否提及数据加密、脱敏、审计日志等安全要求
- **国际化**：如果面向多地区，是否提及多语言、时区等要求

- **测试环境**：opencode v1.2.27 + minimax-m2.5 模型

## 二、各子 Skill 详细评估

### 2.1 req-review（审查）

**测试命令**：`帮我审查一下需求文档 docs/prd.md，看看有什么问题`

**触发情况**：成功触发，正确加载了 `req-review/SKILL.md` 和 `generic-rules.md`

**输出分析**：

| 指标 | 结果 |
|------|------|
| 发现问题数 | 23 个（P0:5, P1:10, P2:8） |
| 问题分类 | 覆盖 ambiguous、missing、missing_constraint、incomplete_flow 4 种类型 |
| 排序 | 按 P0→P1→P2 排序，符合规范 |
| 口头汇总 | 有，格式基本符合预期 |

**优点**：
1. 准确发现了所有故意植入的模糊用词（"合理的"、"尽快"、"一定的"、"及时"）
2. 核心流程缺失（退款流程、支付失败处理）被正确标记为 P0
3. 非功能性缺失（权限、性能、安全、兼容性）均有覆盖
4. 每条缺陷描述清晰，包含具体缺失内容

**问题**：
1. **输出格式不严格符合 issue-schema.md 定义**：缺少 `id`（ISS-001 格式）、`location`、`context`、`status` 字段，用的是自由格式而非规定的表格格式
2. **文件路径解析出错**：尝试读取 `.opencode/skills/req-skill/req-review/reference/business-rules.md`（错误路径），实际应为 `.opencode/skills/req-skill/reference/business-rules.md`
3. **缺少按模块汇总的详细表格**：issue-schema.md 要求有"按模块汇总"部分，输出中有但不够规范
4. **有些 P0 和 P1 边界判定偏宽松**：如"验证码有效期未说明"定为 P0 似乎偏高（实际上有行业默认值可参考）

### 2.2 req-complete（补全）

**测试命令**：`我想补全需求文档 docs/prd.md，对于所有缺陷问题请帮我自动按照行业最佳实践来补全`

**触发情况**：先触发了顶层 `req-skill`，然后自动识别为补全意图

**输出分析**：

| 指标 | 结果 |
|------|------|
| 补全问题数 | 23/23 全部补全 |
| 输出文件 | prd-updated.md + changelog.md |
| 文档版本号 | v1.0（符合模板） |
| 术语表 | 有，包含 8 个术语 |
| TBD 标记 | 0 个（全部自动补全） |

**优点**：
1. **文档结构完全遵循 prd-template.md**：包含元信息块、H2 模块划分、功能描述/业务规则/约束条件/异常处理子章节
2. **补全值合理**：密码 8-20 位、验证码 5 分钟、登录锁定 5 次/30 分钟，均符合行业惯例
3. **完整的权限矩阵**：补充了管理员/普通用户/游客三个角色
4. **订单状态机完整**：补充了"已完成"和"已退款"两个缺失状态，并明确了转换条件
5. **changelog 格式清晰**：每条变更对应原始缺陷，可追溯
6. **正确识别并使用了 review-issues.md**：状态感知功能生效

**问题**：
1. **跳过了逐条提问流程**：由于测试指令要求"自动补全"，skill 没有执行逐条提问的核心流程，无法验证提问策略的效果
2. **changelog 格式与模板有偏差**：缺少 ISS-ID 映射、缺少"已解决/未解决/统计"表格分类
3. **安全性建议有误**：建议"MD5 加盐加密"存储密码，这是过时的做法（应推荐 bcrypt/argon2）
4. **文件路径解析报错**：同样出现了路径解析错误（reference/business-rules.md 的相对路径问题）

## 三、共性问题

### 3.1 文件路径解析问题（高优先级）

三个子 skill 执行时都出现了路径解析错误。根本原因是 SKILL.md 中写的相对路径（如 `reference/business-rules.md`）被模型解析为相对于子 skill 目录（如 `req-review/reference/`）而不是相对于 skill 根目录（`req-skill/reference/`）。

**影响**：虽然模型通过 Glob 搜索自行修复了路径，但增加了不必要的执行步骤，降低了效率。

**建议**：在各子 SKILL.md 中使用更明确的路径说明，如 `../reference/business-rules.md` 或在路径前加 skill 根目录标记。

### 3.2 输出格式遵循度不足（中优先级）

虽然 shared/ 目录下定义了详细的输出模板（issue-schema.md、prd-template.md、testcase-template.md），模型只是"大体遵循"而非"严格遵循"。具体表现为：
- 缺陷清单缺少 ISS-ID 等必填字段
- 测试用例缺少 TC-ID 格式和关联需求字段
- changelog 缺少规定的表格格式

**建议**：在模板文件中增加"必须包含的字段"的强调标记，或提供更明确的 Markdown 输出示例。

### 3.3 模型对 skill 指令的"创造性偏离"（低优先级）

模型在执行 skill 指令时偶尔会偏离规定流程：
- 补全时跳过了提问流程（虽然是用户要求的，但 skill 应提醒用户这不是标准流程）
- 测试用例生成时创造了原始文档中没有的功能（如"分享证书"）
- 安全性建议使用了过时的方案（MD5 加盐）

## 四、改进建议

### 优先级 P0（必须修复）

| # | 问题 | 建议 |
|---|------|------|
| 1 | 文件路径解析错误 | 在子 SKILL.md 中改用相对于 skill 根目录的路径，或增加明确的路径说明："以下所有路径相对于 `req-skill/` 目录" |
| 2 | 输出缺少 ISS-ID | 在 req-review/SKILL.md 的输出步骤中增加明确指令："每条缺陷必须包含 ISS-{三位序号} 格式的 ID" |
| 3 | 测试用例 ID 格式 | 在 req-testcase/SKILL.md 中强调 "用例 ID 必须严格使用 TC-{模块缩写}-{序号} 格式" |

### 优先级 P1（建议修复）

| # | 问题 | 建议 |
|---|------|------|
| 4 | changelog 格式偏差 | 在 req-complete/SKILL.md 中提供一个完整的 changelog 输出示例 |
| 5 | 缺少关联需求字段 | 在 testcase-template.md 中用粗体标记必填字段 |
| 6 | 补全时的领域知识准确性 | 在 reference/ 中增加安全最佳实践参考文档，避免模型用过时方案 |
| 7 | 测试用例与需求矛盾 | 在 req-testcase/SKILL.md 的质量自检步骤中增加："检查生成的用例是否与需求文档内容矛盾" |

### 优先级 P2（可优化）

| # | 问题 | 建议 |
|---|------|------|
| 8 | 模块识别粒度 | 当前按 H2 拆分模块，可考虑支持用户自定义模块边界 |
| 9 | 补全模式选项 | 增加"自动补全"模式作为正式支持的选项，而不仅是逐条提问模式 |
| 10 | 测试用例优先级分配 | P0/P1/P2 的分配逻辑可以更透明，建议在 testcase-template.md 中增加优先级分配指南 |
| 11 | 跨子 skill 的上下文传递 | 当前依赖文件系统（output/ 目录）传递状态，可考虑增加显式的状态摘要传递机制 |
| 12 | business-rules.md 为空的处理 | 当 business-rules.md 中全是注释模板时，skill 应提示用户填写项目特有规则以提升审查精度 |

## 六、总结

req-skill 作为需求文档全流程质量工具，**整体框架设计合理，三个子 skill 的职责划分清晰，串联流程的状态感知机制有效**。在实际测试中，审查功能覆盖度高，补全功能结果实用，测试用例生成覆盖了多种测试类型。

主要的改进方向集中在：
1. **格式规范的强制力**：需要更强的 prompt 工程让模型严格遵循输出模板
2. **路径解析的鲁棒性**：子 skill 中的文件引用路径需要统一规范
3. **领域知识的准确性**：对于安全等专业领域，需要在 reference/ 中提供准确的最佳实践参考

经过上述优化后，该 skill 有潜力成为一个高效可靠的需求文档质量保障工具。

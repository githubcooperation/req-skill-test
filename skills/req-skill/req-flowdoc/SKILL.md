# 流程说明文档生成

> 📁 **路径速查**：skill 内部文件（shared/ 等）相对于 `req-skill/`；output 文件相对于 workspace 根目录；用 `Glob("**/<filename>")` 定位 shared 文件最可靠。

## 职责

从 PRD 中提取业务流程描述和集成接口信息，为每个有意义的业务流程生成一份标准化的流程说明文档。

---

## 执行流程

### 第一步：确认输入文档

> **路径约定**
> - skill 内部文件（shared/、reference/）：相对于 `req-skill/` 根目录
>   - 正确：`shared/flow-doc-template.md`
> - output 文件：相对于 workspace 根目录（eval 工作目录）
>   - 正确：`output/prd-updated.md`、`output/flow-docs/{流程名}.md`
>   - 从本目录访问：`../../output/prd-updated.md`
> - 推荐：用 `Glob("**/flow-doc-template.md")` 等方式定位 shared 文件

按优先级选择输入文档：

1. `output/prd-updated.md` 存在 → 使用此版本（集成接口部分通常在补全后更完整）
2. 不存在 → 询问用户原始 PRD 路径，并提示："建议先完成需求补全流程，流程文档中的接口字段会更完整，TBD 数量会更少。"

同时读取 `shared/flow-doc-template.md`，了解输出格式。

---

### 第二步：扫描并识别业务流程

扫描文档，从以下位置识别可生成流程文档的内容：

**优先扫描位置**（信息密度由高到低）：

1. **第 4 章"集成需求"**：
   - `4.1 数据接口` → 每个对接接口候选为一个流程文档或一个 `4.x` 接口节
   - `4.2 数据初始化` → 如涉及数据导入接口，一并提取

2. **第 3.2.x 需求卡片**：
   - `| 依赖条件 |` 行提到的"上游单据"、"接口"、"外部系统"
   - 业务逻辑描述中包含"调用"、"推送"、"同步"、"接口"、"通知"关键词的段落
   - 处理逻辑中描述系统间数据传递或回调的内容

3. **第 2 章"业务流程"**：
   - 涉及多系统交互的整体流程，可作为顶层流程文档的背景素材

**流程识别规则**：
- 同一方向的数据流（A系统 → B系统）合并为一个流程文档
- 独立的业务节点（核算流程、支付流程、退款流程）各自独立为一个文档
- 仅描述单系统内部操作的**不生成**流程文档（无需接口说明）

**信息完整度评估**（用于提示 TBD 预警）：
- **高**：字段名、数据类型、必填、返回结构在 PRD 中均有明确描述
- **中**：有字段名但缺少数据类型或返回结构
- **低**：接口仅被提及，无任何字段级描述

---

### 第三步：展示识别结果，等待用户确认

在生成任何文件之前，先告知用户：

```
从 PRD 中识别到以下可以生成流程文档的业务流程：

1. {流程名称A}
   来源：{章节位置，如 "第 4.1 节" 或 "3.2.3 需求卡片 > 依赖条件"}
   涉及接口：{N} 个（{接口名称简要列举}）
   信息完整度：{高/中/低}
   {低时补充说明：大部分字段需要标注 TBD，建议先与开发团队确认接口规范后再生成}

2. {流程名称B}
   来源：...
   ...

共识别到 {N} 个流程，是否按以上列表生成？
可以告诉我要跳过哪些，或补充我遗漏的流程。
```

如果未找到任何可识别的流程：

```
在 PRD 中未找到集成接口或跨系统流程的描述。

流程说明文档适用于：
- 需要与外部系统对接的接口（如支付、短信、ERP数据同步）
- 跨模块的数据推送流程
- 第 4 章"集成需求"中描述的接口

如果你的文档包含这类内容，请告诉我具体在哪个章节，我来识别提取。
```

---

### 第四步：逐个生成流程说明文档

严格按 `shared/flow-doc-template.md` 的结构输出，写入 `output/flow-docs/{流程名}.md`。

**文件命名规则**：使用小写英文或拼音，连字符分隔。
示例：`折让核算流程` → `zherang-hesuan.md`，`用户登录接口` → `user-login.md`

**格式规范**：读取 `shared/flow-doc-template.md`（用 Glob 定位），章节结构、字段列名、出参固定结构均以该模板为准。不得省略模板中的必填区块（如出参固定字段 code/success/message/timestamp/result）。

#### TBD 标注约定

- `TBD` — 需要确认，无附加说明
- `TBD（说明原因）` — 带说明的待确认项，帮助读者理解需要向谁确认什么

示例：`TBD（请与开发团队确认该接口是同步调用还是异步回调）`

**写完后自检（逐条确认）：**
1. ✅ 每个接口的出参固定字段（code / success / message / timestamp / result）已完整复制，不得省略
2. ✅ 场景表中的接口编号（4.1, 4.2...）与章节 4.x 标题一一对应，无错位
3. ✅ 未确定字段标注 `TBD（请补充）`，不留空白单元格
4. ✅ 无内容的可选区块已删除（不保留空标题）
5. ✅ 文件名使用小写英文/拼音 + 连字符格式（如 `user-login.md`、`zherang-hesuan.md`）

---

### 第五步：输出与汇总

所有文档生成完毕后：

1. 确认所有文件已写入 `output/flow-docs/`
2. 统计 TBD 数量：计算所有生成文件中值为 `TBD` 或以 `TBD（` 开头的表格单元格总数
3. 在对话中给出汇总：

```
流程文档生成完成！

| 流程名称 | 文件 | 接口数 | TBD 数量 | 信息完整度 |
|---------|------|--------|---------|-----------|
| {流程A} | {文件名}.md | {N} | {N} | 高/中/低 |
| {流程B} | {文件名}.md | {N} | {N} | 高/中/低 |

所有文件已写入 output/flow-docs/ 目录。
```

如果 TBD 数量 > 0，在汇总后追加：`共有 {N} 个 TBD 字段，建议在技术评审时与开发团队逐一填写。`

---

## 已有流程文档的处理

如果 `output/flow-docs/` 目录下已有文件：

```
output/flow-docs/ 目录下已有以下流程文档：
{列出文件名}

请选择：
A) 全部覆盖，重新生成
B) 只生成缺少的
C) 指定要重新生成哪些（请列出文件名）
D) 取消，保留现有文件
```

选择 A 或 C 覆盖已有文件时，将原文档头部的版本号递增（如 `v1.0` → `v1.1`），并更新"最后更新"日期为当前日期。

# 提问策略

本文件定义 req-complete 在逐条提问时的排序逻辑、跳过规则和特殊情况处理。

## 排序规则

### 主排序：严重程度
P0 → P1 → P2，阻塞性问题优先解决。

### 次排序：模块聚合
同一严重程度内，按模块分组。同一模块的问题连续提问，这样用户的思维停留在同一个功能上下文中，回答质量更高。

模块间的顺序按文档中的出现顺序排列。

### 三级排序：依赖关系
如果问题之间存在依赖关系（A 的答案会影响 B 的提问方式），A 排在 B 前面。

常见依赖关系模式：
- "是否需要此功能" → "此功能的具体规则"
- "角色权限定义" → "各角色的操作流程"
- "数据格式定义" → "数据的校验规则"
- "状态列表" → "状态之间的转换条件"

## 跳过规则

以下情况自动跳过某个问题，不向用户提问：

### 被动解决
用户回答了问题 A 后，如果答案同时覆盖了问题 B 的缺口，跳过 B。

判断逻辑：
- B 的 `type` 是 `missing`，且 A 的回答中包含了 B 所缺失的信息
- B 的 `type` 是 `ambiguous`，且 A 的回答中明确了 B 所模糊的概念

跳过时告知用户：
```
（跳过第 {N} 题：{问题描述}。你刚才的回答已经覆盖了这个问题。）
```

### 已在文档中存在
二次审查时，某些问题可能已在上一轮补全中解决。如果 `review-issues.md` 中该条的 `status` 已经是 `resolved`，跳过。

### 用户主动跳过
用户在提问前说"P2 的问题全跳过"或"这个模块不用问了"，尊重用户意愿，批量标记为 TBD 或跳过。

## TBD 处理

### 用户说"不确定"或"待定"时
1. 在文档中插入 `[TBD-{ISS-ID}]` 标记
2. 在 changelog 中记录为"未解决"，保留原因
3. 在 `review-issues.md` 中将 status 改为 `tbd`
4. 不追问，不施压，继续下一个问题

### TBD 的表述识别
以下用户回答均视为 TBD：
- "不确定"、"不知道"、"待定"、"TBD"
- "这个要问一下开发/设计/业务"
- "回头再说"、"先跳过"、"下次再确认"
- "我不是很清楚这个"

### TBD 不是终点
在所有问题处理完毕的口头汇总中，列出所有 TBD 项并提示用户后续可以再跑一轮补全。

## 回答冲突处理

### 与文档已有内容矛盾
用户的回答与文档中其他章节的描述不一致时：

```
注意：你的回答（{用户回答}）与文档中 {定位} 的描述（{原文内容}）不一致。
请确认以哪个为准？
A) 以你刚才的回答为准（我会同步更新相关章节）
B) 以文档原有内容为准（本条缺陷标记为已解决）
C) 两处都不对，应该是：______
```

### 与前面的回答矛盾
用户本轮的回答之间出现矛盾时，指出矛盾并请求确认，但语气保持中性，不暗示哪个是"对的"。

## 提问话术指南

### 给出候选选项时
```
[3/15] [P0] 用户注册

验证码的有效期是多长时间？

常见选择：
A) 60秒
B) 120秒（行业最常见）
C) 300秒
D) 其他（请说明）
```

### 是/否类问题
```
[5/15] [P1] 订单管理

用户是否可以在支付后取消订单？
A) 可以（请说明时间限制和退款规则）
B) 不可以
C) 有条件地可以（请说明条件）
```

### 开放性问题
```
[8/15] [P1] 支付流程

支付失败后系统如何处理？

以下是常见的处理方式供参考：
- 提示用户重试，保留当前订单
- 自动取消订单，释放库存
- 保留订单 N 分钟，超时后自动取消

你们的方案是？
```

### 需要上下文的问题
```
[10/15] [P1] 权限管理

原文中提到"管理员可以管理所有用户"，但没有说明具体可以执行哪些操作。

管理员对用户可以做哪些事？（可多选）
A) 查看用户列表和详情
B) 编辑用户信息
C) 禁用/启用用户账号
D) 删除用户
E) 重置用户密码
F) 其他（请说明）
```

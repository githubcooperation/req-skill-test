# 需求文档补全

> 📁 **路径速查**：skill 内部文件（shared/ 等）相对于 `req-skill/`；output 文件相对于 workspace 根目录；用 `Glob("**/<filename>")` 定位 shared 文件最可靠。

## 职责

接受任意格式的输入（采访稿、会议记录、零散需求点、不规范 PRD、已结构化 PRD），
先将内容整理为标准 PRD 结构（格式规范已内嵌在第二步中，无需读取外部模板文件），
再通过逐条提问补全缺失信息，最终输出可以直接驱动开发、测试和线框图生成的完整需求文档。

## 执行流程

### 第一步：确认输入

> **路径约定**
> - skill 内部文件（shared/、reference/）：相对于 `req-skill/` 根目录
>   - 正确：`req-complete/question-strategy.md`、`reference/security-best-practices.md`、`shared/prd-template.md`
> - output 文件：相对于 workspace 根目录（eval 工作目录）
>   - 正确：`output/prd-updated.md`、`output/review-issues.md`、`output/changelog.md`
>   - 从本目录访问：`../../output/prd-updated.md`
> - 推荐：用 `Glob("**/prd-template.md")` 等方式定位 shared 文件，无需计算层级

确定两个输入来源：

1. **原始材料**：
   - 如果 `output/prd-updated.md` 存在 → 使用此版本（上一轮的输出，在此基础上继续）
   - 否则 → 使用用户指定的文件路径；如果用户未提供路径，主动询问："请告诉我输入材料的文件路径（可以是需求文档、采访稿、会议记录等任意格式）。"

2. **缺陷清单**：检查 `output/review-issues.md` 是否存在
   - 存在 → 读取，作为后续提问的问题来源之一
   - 不存在 → 不报错，后续从结构化转换阶段自行识别缺失项

**输入类型检测**：读取输入材料后，**立即输出以下确认行**（防止处理错误文档）：

```
正在处理：《{系统/文档名称}》，输入类型：{A/B/C 类}
```

然后判断属于以下哪类：

- **A 类（已符合模板）**：文档中已有 `#### 3.2.` 标题结构 → 直接跳到第三步
- **B 类（不规范 PRD）**：有段落/章节划分，有需求描述，但结构不符合模板 → 进入第二步做结构映射
- **C 类（原始素材）**：采访稿、会议记录、对话内容、零散需求点，无明显文档结构 → 进入第二步做完整结构提取

---

### 第二步：结构化转换（B 类、C 类输入执行此步，A 类跳过）

此步骤分为两个阶段，必须按顺序执行：

---

**阶段 2A：自由提取（内容优先，不限格式）**

从输入中自由提取以下内容，列成清单即可，不需要考虑任何格式：

- 系统名称
- 背景和目标
- 功能模块列表（每个模块的名称、涉及的用户角色）
- 每个模块的字段/信息项
- 每个模块的业务规则和约束
- 异常处理和边界情况
- 集成需求（如涉及外部系统）
- 无法从输入确定的内容（标注为 TBD）

---

**阶段 2B：逐模块输出标准格式文档**

用 2A 提取的内容生成完整的需求文档，遵守以下格式规则：

**格式规则**（每条都是硬性要求）：
1. 文档以 `# {系统名} 需求说明书` 开头，后跟版本、日期元信息
2. 功能模块**只能**使用 `#### 3.2.N` 作为标题（N 从 1 开始递增）
   - ✅ 正确：`#### 3.2.1 员工预约`、`#### 3.2.2 HR管理`
   - ❌ 禁止：`## 功能模块`、`### 员工端`、`#### 1.`、`#### 2.1.1`、`#### F-001`
3. 每个 `#### 3.2.N` 块末尾**必须**有独立一行，内容是以下7个字的字面量：

   `{低保真原型图}`

   **重要**：`{低保真原型图}` 是**字面文字**（literal string），不是占位符，不要替换为其他内容，不要删除，必须原样输出这7个字

**格式权威来源**：`shared/prd-template.md`（用 Glob 定位）。以下示例展示关键格式规则，以 shared/prd-template.md 中的完整结构为准。

**正确格式示例**（以一个假设模块说明格式，非本次内容）：

```
# 示例系统 需求说明书

- **文档版本**：v1.0
- **创建时间**：2026-01-01
- **编写人**：AI 整理

## 1. 项目概述

### 1.1 项目背景

背景描述文字。

### 1.2 实施范围

范围描述文字。

## 2. 业务流程

TBD

## 3. 业务需求描述

### 3.1 功能需求整理

| 模块 | 功能 | 操作端 |
|------|------|--------|
| 员工预约 | 在线预约体检 | 移动端 |

### 3.2 功能需求详述

#### 3.2.1 员工预约

**业务场景**：员工通过移动端选择套餐和时段提交预约。

**页面字段**：

| 名称 | 是否必需 | 字段说明 |
|------|----------|---------|
| 姓名 | 是 | 从HR系统同步，只读 |
| 体检套餐 | 是 | 三选一 |

**业务规则**：

1. 每年限1次预约

{低保真原型图}

---

## 4. 集成需求

如无集成需求则删除本章。
```

现在用 2A 提取的**实际内容**，按上述格式输出完整文档。先输出文档头部，再逐个输出 `#### 3.2.1`、`#### 3.2.2`...，每个模块末尾必须有 `{低保真原型图}` 这一行。

4. **告知用户结构识别结果**：

```
已将输入内容整理为标准 PRD 结构：

识别到 {N} 个功能模块：
1. {模块A}（{已提取字段数} 个字段已填写，{TBD数} 个待确认）
2. {模块B}（...）
...

共有 {TBD总数} 个字段需要确认，接下来我会逐一询问。
```

### 第三步：规划提问顺序

读取 `req-complete/question-strategy.md`（即 `req-skill/req-complete/question-strategy.md`）获取提问策略。核心原则：

- 如果有 `review-issues.md`，将其中的 P0 问题优先处理
- 结构化转换产生的 TBD 项按模块顺序处理，同一模块连续问
- 预判依赖关系：如果问题 A 的答案可能影响问题 B，先问 A

在开始提问之前，先告诉用户整体情况：

```
共有 {总数} 个问题需要确认：
- P0（阻塞性）：{数量} 个
- P1（重要）：{数量} 个
- P2（改进）：{数量} 个

我会逐个问你，尽量给出候选选项方便你选择。
如果某个问题你暂时无法确认，可以说"待定"，我会标记为 TBD。
开始吧？
```

---

### 第四步：逐条提问

每次只问一个问题。提问格式：

```
[{当前序号}/{总数}] [{严重度}] {模块名}

{问题描述}

{候选选项，如果适用}
```

**候选选项的生成规则**：
- 对于有限选项的问题（如"验证码有效期"），给出 2-4 个业界常见值，如："A) 60秒  B) 120秒  C) 300秒  D) 其他（请说明）"
- 对于是/否类问题，直接给出"是/否"选项
- 对于开放性问题（如"异常时的处理方式"），给出 2-3 个常见方案作为参考，同时允许用户自由回答
- 无法给出候选选项时，直接提问

**收到用户回答后**：
- 明确回答 → 记录答案，在内存中更新文档内容，继续下一个问题
- 回答"待定"/"不确定"/"TBD" → 在文档中标记 `[TBD-{ISS-ID}]`，在 changelog 中记录为未解决项，继续下一个问题
- 回答引出了新信息 → 判断是否导致其他缺陷自动解决，如果是则跳过已解决的问题并告知用户
- 回答与已有信息矛盾 → 指出矛盾，请用户确认以哪个为准

---

### 第五步：输出文件

所有问题处理完毕后，生成两个文件：

按顺序执行以下三个写文件操作：

**先执行：更新缺陷状态（如果 output/review-issues.md 存在）**

直接修改 `output/review-issues.md`（不创建新文件），将本轮处理过的条目 status 改为对应值：
- 已解决 → `resolved`
- 标记待定 → `tbd`
- 未处理 → 保持 `open`

**再执行：**

**1. 补全后的需求文档** → `output/prd-updated.md`

**格式要求**（不需要另外读取模板文件，格式规范已在第二步阶段 2B 的示例中说明）：
- 如果是在上一版基础上补全，版本号递增（如 v1.0 → v1.1）
- 每个功能模块必须使用 `#### 3.2.x` 标题格式
- 每个 `#### 3.2.x` 块末尾必须有独立一行：`{低保真原型图}`（字面量，原样写入）
- 仍未确认的字段保留 `[TBD-{ISS-ID}]` 标记

**1.5 写入 prd-updated.md 后，立即运行以下修复脚本（必须在写 changelog.md 之前执行）**

```bash
python3 -c "
lines = open('output/prd-updated.md').readlines()
out, buf = [], []
for ln in lines:
    if ln.strip() == '---':
        if any(l.startswith('#') for l in buf) and '{低保真原型图}' not in ''.join(buf):
            out += ['\n', '{低保真原型图}\n', '\n']
        buf = []
    else:
        buf.append(ln)
    out.append(ln)
open('output/prd-updated.md', 'w').writelines(out)
print('占位符数:', open('output/prd-updated.md').read().count('{低保真原型图}'))
"
```

此脚本会自动扫描文档中每个内容章节（含 `#` 标题），在缺少 `{低保真原型图}` 的章节末尾（`---` 之前）补充该占位符。运行后确认输出"占位符数: N"（N > 0）再继续。

**2. 变更记录** → `output/changelog.md`（追加模式）

changelog **必须严格**使用以下格式，包含 ISS-ID 映射和三个分类表格（已解决/未解决/统计）：

```markdown
## 轮次 {N} - {YYYY-MM-DD HH:mm}

### 已解决
| ISS-ID | 模块 | 描述 | 确认结果 |
|--------|------|------|----------|
| ISS-001 | 用户注册 | 验证码有效期 | 120秒 |
| ISS-002 | 订单管理 | 取消时限 | 支付后30分钟内 |

### 未解决（TBD）
| ISS-ID | 模块 | 描述 | 原因 |
|--------|------|------|------|
| ISS-005 | 支付 | 退款到账时间 | 待确认第三方支付渠道的处理时效 |

### 自动跳过
| ISS-ID | 模块 | 描述 | 被哪条回答覆盖 |
|--------|------|------|----------------|
| ISS-003 | 用户注册 | 密码规则 | 由 ISS-001 的回答覆盖 |

### 统计
- 本轮处理：{总数} 个问题
- 已解决：{数量}
- 标记为 TBD：{数量}
- 自动跳过（因其他回答已覆盖）：{数量}
```

> **注意**：changelog 中每条记录**必须**包含 ISS-ID，用于追溯到原始缺陷清单。如果输入的缺陷清单没有 ISS-ID（例如上游审查未按规范输出），则在 changelog 中自行编号为 `FIX-001`、`FIX-002` 等。

**3. 写完文件后自检并修复**：

写入 `output/prd-updated.md` 后，执行以下检查，发现问题立即修复（edit 文件，不要重写整个文件）：

```
[ ] 检查1：prd-updated.md 中每个 #### 3.2.x 卡片都有 {低保真原型图} 这一行
    → 如果缺少，在该卡片末尾（---之前）插入 {低保真原型图}
[ ] 检查2：prd-updated.md 功能卡片均使用 #### 3.2.x 编号，无自定义标题
    → 如果有 #### 2.1.x 或 ## 模块名 等非标准格式，需重写该部分
[ ] 检查3：changelog.md 包含已解决/未解决/统计三个表格
[ ] 检查4：若 review-issues.md 存在，已在写 prd-updated.md 之前更新了 status
```

## 补全模式

req-complete 支持两种模式：

### 模式一：逐条提问（默认）
按上述第四步的流程逐条向用户提问，适合用户有时间参与且希望精确控制每个细节的场景。

### 模式二：自动补全
当用户明确说"自动补全"、"按行业最佳实践补全"、"不用问我直接补"等意图时，跳过逐条提问流程，自动按行业最佳实践填充所有缺陷项。

**自动补全模式的执行规则**：
1. 在开始前告知用户："将按行业最佳实践自动补全所有 {总数} 个问题，不会逐条提问。如果某些项需要项目特定信息，会标记为 TBD。"
2. 对于有明确行业标准的项（如验证码有效期、密码长度），直接使用行业最佳实践值
3. 对于安全相关的技术方案，**必须**参考 `reference/security-best-practices.md`（如果存在），**禁止**推荐已过时的方案（如 MD5 加盐）
4. 对于高度项目特定、无法推断的项（如具体业务金额、特殊审批流程），标记为 TBD
5. changelog 中明确标注"（自动补全）"，与手动确认的记录区分
6. 完成后提醒用户："已自动补全 {数量} 项，其中 {TBD数量} 项因需要项目特定信息已标记为 TBD，建议逐一检查补全结果。"

**注意**：自动补全模式同样会执行第二步的结构化转换（如果输入为 B 类或 C 类）。

## 提问原则（适用于逐条提问模式）

- **每次只问一个问题**。不要一次抛出多个问题让用户回答
- **给出上下文**。提问时引用原文的相关段落，让用户知道是针对哪里
- **尊重用户的"待定"**。不追问、不施压，标记 TBD 后继续
- **不替用户做决定**。候选选项是参考，不是默认值
- **回答有歧义时确认**。宁可多确认一次，不要自行推断

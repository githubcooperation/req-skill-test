# 需求文档补全

## 职责

基于审查缺陷清单（或直接基于原始文档），通过逐条向用户提问的方式收集信息，将需求文档补全到可以驱动开发和测试的程度。

## 执行流程

### 第一步：确认输入

需要两份输入文件：

1. **缺陷清单**：检查 `output/review-issues.md` 是否存在
   - 存在 → 读取并解析为待处理的问题列表
   - 不存在 → 提示用户："还没有审查结果，要先帮你审查一遍吗？还是你直接告诉我要补充哪些内容？"

2. **需求文档**：
   - 如果 `output/prd-updated.md` 存在 → 使用此版本（上一轮补全的结果）
   - 否则 → 使用用户指定的原始文档路径

### 第二步：规划提问顺序

> **路径说明**：本文件中所有文件路径均相对于 skill 根目录（即 `req-skill/`），而非当前子目录 `req-skill/req-complete/`。例如 `shared/prd-template.md` 的完整路径是 `req-skill/shared/prd-template.md`，`reference/business-rules.md` 的完整路径是 `req-skill/reference/business-rules.md`。

读取 `req-complete/question-strategy.md`（即 `req-skill/req-complete/question-strategy.md`）获取提问策略。核心原则：

- 按严重程度排序：P0 先问，P1 其次，P2 最后
- 同一模块的问题连续问，减少上下文跳跃
- 预判依赖关系：如果问题 A 的答案可能影响问题 B，先问 A

在开始提问之前，先告诉用户整体情况：

```
共有 {总数} 个问题需要确认：
- P0（阻塞性）：{数量} 个
- P1（重要）：{数量} 个
- P2（改进）：{数量} 个

我会逐个问你，尽量给出候选选项方便你选择。
如果某个问题你暂时无法确认，可以说"待定"，我会标记为 TBD。
开始吧？
```

### 第三步：逐条提问

每次只问一个问题。提问格式：

```
[{当前序号}/{总数}] [{严重度}] {模块名}

{问题描述}

{候选选项，如果适用}
```

**候选选项的生成规则**：
- 对于有限选项的问题（如"验证码有效期"），给出 2-4 个业界常见值，如："A) 60秒  B) 120秒  C) 300秒  D) 其他（请说明）"
- 对于是/否类问题，直接给出"是/否"选项
- 对于开放性问题（如"异常时的处理方式"），给出 2-3 个常见方案作为参考，同时允许用户自由回答
- 无法给出候选选项时，直接提问

**收到用户回答后**：
- 明确回答 → 记录答案，在内存中更新文档内容，继续下一个问题
- 回答"待定"/"不确定"/"TBD" → 在文档中标记 `[TBD-{ISS-ID}]`，在 changelog 中记录为未解决项，继续下一个问题
- 回答引出了新信息 → 判断是否导致其他缺陷自动解决，如果是则跳过已解决的问题并告知用户
- 回答与已有信息矛盾 → 指出矛盾，请用户确认以哪个为准

### 第四步：输出文件

所有问题处理完毕后，生成两个文件：

**1. 补全后的需求文档** → `output/prd-updated.md`

按 `shared/prd-template.md` 的格式输出。如果是在上一版基础上补全，版本号递增。

**2. 变更记录** → `output/changelog.md`（追加模式）

changelog **必须严格**使用以下格式，包含 ISS-ID 映射和三个分类表格（已解决/未解决/统计）：

```markdown
## 轮次 {N} - {YYYY-MM-DD HH:mm}

### 已解决
| ISS-ID | 模块 | 描述 | 确认结果 |
|--------|------|------|----------|
| ISS-001 | 用户注册 | 验证码有效期 | 120秒 |
| ISS-002 | 订单管理 | 取消时限 | 支付后30分钟内 |

### 未解决（TBD）
| ISS-ID | 模块 | 描述 | 原因 |
|--------|------|------|------|
| ISS-005 | 支付 | 退款到账时间 | 待确认第三方支付渠道的处理时效 |

### 自动跳过
| ISS-ID | 模块 | 描述 | 被哪条回答覆盖 |
|--------|------|------|----------------|
| ISS-003 | 用户注册 | 密码规则 | 由 ISS-001 的回答覆盖 |

### 统计
- 本轮处理：{总数} 个问题
- 已解决：{数量}
- 标记为 TBD：{数量}
- 自动跳过（因其他回答已覆盖）：{数量}
```

> **注意**：changelog 中每条记录**必须**包含 ISS-ID，用于追溯到原始缺陷清单。如果输入的缺陷清单没有 ISS-ID（例如上游审查未按规范输出），则在 changelog 中自行编号为 `FIX-001`、`FIX-002` 等。

同时更新 `output/review-issues.md` 中对应条目的 `status` 字段。

### 第五步：口头汇总

```
补全完成！

- 处理了 {总数} 个问题，其中 {已解决数} 个已确认，{TBD数} 个标记为待定
- 更新后的文档已写入 output/prd-updated.md（版本 v{N}.0）
- 变更记录已追加到 output/changelog.md

{如果有 TBD} 还有 {TBD数} 个待定项，后续确认后可以再跑一轮补全。
要继续生成测试用例吗？
```

## 补全模式

req-complete 支持两种模式：

### 模式一：逐条提问（默认）
按上述第三步的流程逐条向用户提问，适合用户有时间参与且希望精确控制每个细节的场景。

### 模式二：自动补全
当用户明确说"自动补全"、"按行业最佳实践补全"、"不用问我直接补"等意图时，跳过逐条提问流程，自动按行业最佳实践填充所有缺陷项。

**自动补全模式的执行规则**：
1. 在开始前告知用户："将按行业最佳实践自动补全所有 {总数} 个问题，不会逐条提问。如果某些项需要项目特定信息，会标记为 TBD。"
2. 对于有明确行业标准的项（如验证码有效期、密码长度），直接使用行业最佳实践值
3. 对于安全相关的技术方案，**必须**参考 `reference/security-best-practices.md`（如果存在），**禁止**推荐已过时的方案（如 MD5 加盐）
4. 对于高度项目特定、无法推断的项（如具体业务金额、特殊审批流程），标记为 TBD
5. changelog 中明确标注"（自动补全）"，与手动确认的记录区分
6. 完成后提醒用户："已自动补全 {数量} 项，其中 {TBD数量} 项因需要项目特定信息已标记为 TBD，建议逐一检查补全结果。"

## 提问原则（适用于逐条提问模式）

- **每次只问一个问题**。不要一次抛出多个问题让用户回答
- **给出上下文**。提问时引用原文的相关段落，让用户知道是针对哪里
- **尊重用户的"待定"**。不追问、不施压，标记 TBD 后继续
- **不替用户做决定**。候选选项是参考，不是默认值
- **回答有歧义时确认**。宁可多确认一次，不要自行推断

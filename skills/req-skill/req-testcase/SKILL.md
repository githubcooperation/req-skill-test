# 测试用例生成

> 📁 **路径速查**：skill 内部文件（shared/ 等）相对于 `req-skill/`；output 文件相对于 workspace 根目录；用 `Glob("**/<filename>")` 定位 shared 文件最可靠。

## 职责

读取补全后的需求文档，识别模块划分，对照测试类型清单，逐模块生成结构化的测试用例。

## 执行流程

### 第一步：确认输入文档

按优先级选择输入文档：

1. `output/prd-updated.md` 存在 → 使用此版本
2. 不存在 → 使用用户指定的原始文档路径，并提示：
   ```
   注意：当前使用的是未补全的原始文档。
   建议先运行审查和补全流程，测试用例的覆盖度和准确性会更高。
   继续使用原始文档生成吗？
   ```

### 第二步：前置校验

在生成测试用例之前，执行以下检查：

**TBD 扫描**：
扫描文档中的 `[TBD-` 标记。如果存在：
```
文档中有 {数量} 个待定项（TBD），涉及以下模块：
{列出模块名和 TBD 数量}

这些待定项对应的功能点，我会生成"基础版"测试用例（覆盖已明确的部分），
并在用例中标注"待 TBD 解决后补充"。继续吗？
```

**矛盾快检**：
快速扫描文档内容，如果发现明显的前后矛盾（同一字段描述不一致等），指出并建议用户先解决。这一步不需要做完整审查，只是一个轻量的 gate check。

### 第三步：识别模块

> **模块识别规则**（与 `shared/glossary.md` 保持一致）：
> - 优先识别 `#### 3.2.N` 作为需求卡片（req-complete 标准输出格式）
> - 分文件粒度 = 卡片的上一级（即 `### 3.2` 或 `## 3` 模块），一个模块一个文件
> - 若文档无 `#### 3.2.N` 格式，则以 `### N.x` 作为模块，以 `#### N.x.y` 子章节作为卡片

按以上规则识别模块。每个识别到的模块对应一个输出文件。

如果文档完全没有使用标题划分结构，尝试以下降级策略：
1. 按 H2 标题划分
2. 按 H1 标题划分
3. 按明显的功能边界（如空行分隔的大段落）划分
4. 如果仍然无法划分，将整个文档视为一个模块

告知用户识别到的模块列表：
```
识别到以下 {数量} 个模块：
1. {模块A}
2. {模块B}
3. {模块C}

要调整吗？还是按这个划分开始生成？
```

### 第四步：加载测试类型清单

> **路径约定**
> - skill 内部文件（shared/、reference/）：相对于 `req-skill/` 根目录
>   - 正确：`reference/testcase-checklist.md`、`shared/testcase-template.md`
> - output 文件：相对于 workspace 根目录（eval 工作目录）
>   - 正确：`output/prd-updated.md`、`output/test-cases/{模块名}.md`
>   - 从本目录访问：`../../output/prd-updated.md`
> - 推荐：用 `Glob("**/testcase-template.md")` 等方式定位 shared 文件

读取 `reference/testcase-checklist.md`（即 `req-skill/reference/testcase-checklist.md`），获取需要覆盖的测试类型列表。

对每个模块，判断哪些测试类型适用：
- 不是所有类型都适用于所有模块（例如"状态流转"不适用于没有状态变化的模块）
- 跳过不适用的类型，不生成无意义的用例

### 第五步：逐模块生成

对每个模块：

1. 提取该模块的所有需求信息（功能描述、业务规则、状态流转、约束条件、异常处理）
2. 对照每个适用的测试类型，生成对应的测试用例
3. **严格**按 `shared/testcase-template.md` 的格式输出，特别注意：
   - 用例 ID **必须**使用 `TC-{模块缩写}-{三位序号}` 格式（如 `TC-UR-001`），**禁止**使用 1.1、1.2 等编号
   - 每个用例**必须**包含"关联需求"字段，格式固定为三级：`{模块名} > {卡片名} > {具体规则描述}`
     - ✅ `员工端 > 体检预约 > 每日限额规则：每个时段最多30人`
     - ✅ `HR管理后台 > 数据导出 > 导出格式：仅支持Excel`
     - ❌ `体检预约模块`（粒度不足，只有一级）
     - ❌ `3.2.1`（无描述文字）
     - ❌ `见需求文档`（无意义引用）
   - 每个文件末尾**必须**包含"按测试类型汇总表"（而不仅是 P0/P1/P2 统计）
4. 写入 `output/test-cases/{模块文件名}.md`（文件命名规则：使用小写英文或拼音，空格替换为连字符，示例：`user-register.md`、`order-management.md`）

每个模块写入完毕后，在对话中输出简短进度确认：
```
已生成模块 {当前序号}/{总数}：{模块名} → {文件名}.md（{用例数} 个用例）
```

生成时遵循以下原则：

**格式示例（两条用例）：**

### TC-UR-001：正常注册流程

- **类型**：正向流程
- **优先级**：P0
- **前置条件**：用户未注册，位于注册页面
- **测试步骤**：
  1. 输入手机号 13800138000
  2. 点击"获取验证码"，输入收到的6位验证码
  3. 输入密码 Aa1@5678，点击"注册"
- **预期结果**：页面显示"注册成功"，3秒后跳转到首页
- **关联需求**：用户注册 > 注册流程 > 正向注册路径

---

### TC-UR-002：密码长度为7位（低于下限）

- **类型**：边界值
- **优先级**：P1
- **前置条件**：用户位于注册页面
- **测试步骤**：
  1. 输入密码 Aa1@567（7位，低于最小长度8位）
  2. 点击"注册"
- **预期结果**：页面显示"密码长度不符合要求"错误提示，不跳转
- **关联需求**：用户注册 > 业务规则 > 密码长度：8-20位

---

**测试步骤要具体**：
- ✗ "输入有效数据"
- ✓ "在手机号输入框中输入 13800138000"

**预期结果要可验证**：
- ✗ "系统正常处理"
- ✓ "页面显示'注册成功'，并在3秒后跳转到首页"

**每个业务规则至少一个用例**：
需求文档中列出的每条业务规则，至少对应一个正向测试用例验证其生效。

**边界值要覆盖两端**：
如果规则是"密码长度 6-20 位"，至少生成：5位（低于下限）、6位（下限）、20位（上限）、21位（超过上限）四个边界用例。

**TBD 相关的用例标注**：
如果某个功能点包含 TBD 标记，在生成的用例中注明：
```
- **备注**：此用例依赖 [TBD-ISS-003]，待确认后需更新预期结果
```

### 第六步：输出与汇总

所有模块生成完毕后：

1. 确认所有文件已写入 `output/test-cases/`
2. 在对话中给出汇总：

```
测试用例生成完成！

| 模块 | 文件 | 用例数 | P0 | P1 | P2 |
|------|------|--------|----|----|-----|
| 用户注册 | user-register.md | 14 | 3 | 8 | 3 |
| 订单管理 | order-management.md | 22 | 5 | 12 | 5 |
| ... | | | | | |
| **合计** | | **{总数}** | | | |

{如果有TBD} 其中 {数量} 个用例受 TBD 影响，已标注待补充。
所有文件已写入 output/test-cases/ 目录。
```

## 已有测试用例的处理

如果 `output/test-cases/` 目录下已有文件：

```
output/test-cases/ 目录下已有以下测试用例文件：
{列出文件名}

请选择：
A) 全部覆盖重新生成
B) 只生成缺少的模块（保留已有文件）
C) 指定重新生成哪些模块（请列出模块名）
D) 取消，保留全部现有文件
```

## 质量自检（必须执行）

每个模块生成完毕后，**必须**逐一自检以下项目：

1. **需求一致性检查（最重要）**：逐条检查生成的用例是否与需求文档的明确描述矛盾。例如，如果需求文档写了"评价发布后不可修改或删除"，则**禁止**生成"用户修改评价"的用例。发现矛盾时删除该用例。
2. **不得凭空创造功能**：只基于需求文档中明确描述的功能生成用例，不得为文档中未提及的功能（如"分享证书到社交媒体"）生成用例。
3. 每条业务规则是否至少有一个对应用例
4. 正向流程和异常流程的比例是否合理（异常用例通常应多于正向用例）
5. 用例之间是否有重复
6. 测试步骤中是否有模糊描述
7. 用例 ID 是否严格遵循 `TC-{模块缩写}-{三位序号}` 格式
8. 每个用例是否都有"关联需求"字段，且格式为三级（模块名 > 卡片名 > 具体规则描述）
9. 是否覆盖了 `reference/testcase-checklist.md` 中所有适用的测试类型（包括兼容性测试和回退测试）
10. 汇总表中各类型的数字与正文中实际用例数完全一致（逐行数过）
11. 文件名使用小写英文/拼音 + 连字符格式，与模块名对应（如 `user-register.md`、`order-management.md`）

# 安全最佳实践参考

本文件供 req-complete 在自动补全安全相关需求时参考，避免推荐过时或不安全的方案。

## 密码存储

- **推荐**：bcrypt（cost factor ≥ 10）、Argon2id、scrypt
- **禁止**：MD5（含加盐）、SHA-1、SHA-256 直接哈希（即使加盐也不推荐用于密码存储）
- **原因**：MD5/SHA 系列是通用哈希函数，计算速度太快，容易被暴力破解；bcrypt/Argon2 是专为密码设计的慢哈希函数

## 密码策略

- 最小长度：8 位（NIST 800-63B 推荐最少 8 位）
- 最大长度：不少于 64 位（避免限制过短）
- 复杂度：建议要求大小写字母 + 数字 + 特殊字符，但不应过于复杂导致用户体验差
- 常见弱密码检查：建议对照常见弱密码字典校验

## 验证码

- 有效期：60-300 秒（常见选择 120 秒）
- 长度：6 位数字
- 频率限制：同一手机号 60 秒内不可重复发送
- 每日上限：同一手机号每日不超过 10 次
- 验证码错误次数限制：连续错误 5 次后锁定 30 分钟

## 会话管理

- 会话令牌使用安全随机数生成（≥ 128 位）
- 传输过程使用 HTTPS
- Cookie 设置 HttpOnly、Secure、SameSite 属性
- 会话超时：活跃超时 30 分钟，绝对超时 24 小时
- 登出后立即销毁服务端会话

## 登录安全

- 登录失败锁定：连续 5 次失败后锁定账号 30 分钟
- 登录失败提示：统一提示"用户名或密码错误"，不透露具体是哪个错误
- 敏感操作二次验证：修改密码、绑定手机等需要再次验证身份

## 数据传输

- 所有接口使用 HTTPS（TLS 1.2+）
- 敏感数据（密码、支付信息）禁止出现在 URL 参数中
- API 响应中不返回密码哈希等敏感字段

## 输入校验

- 所有用户输入在服务端进行校验（不仅依赖前端校验）
- 防 SQL 注入：使用参数化查询
- 防 XSS：对输出进行 HTML 编码/转义
- 防 CSRF：使用 CSRF Token 或 SameSite Cookie

## 支付安全

- 金额以分为单位在服务端计算，避免浮点精度问题
- 订单金额在服务端校验，不信任前端传递的金额
- 支付结果以回调通知为准，不以前端跳转为准
- 支付接口做幂等性设计，防止重复扣款

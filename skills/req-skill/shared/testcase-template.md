# 测试用例输出模板

本文件定义 req-testcase 输出的每个模块测试用例文件的格式。

## 文件命名

每个模块输出一个独立文件到 `output/test-cases/` 目录，文件名为模块名的小写拼音或英文，用连字符分隔：

- `用户注册` → `user-register.md`
- `订单管理` → `order-management.md`
- `支付流程` → `payment-flow.md`

如果模块名本身是英文则直接使用。

## 文件结构

```markdown
# {模块名} - 测试用例

- **需求文档版本**：{prd-updated.md 的版本号}
- **生成时间**：{时间戳}
- **用例总数**：{总数}
- **覆盖类型**：{本模块涉及的测试类型列表}

## 用例列表

### TC-{模块缩写}-001：{用例标题}

- **类型**：{测试类型，如正向流程/异常流程/边界值等}
- **优先级**：{P0/P1/P2}
- **前置条件**：{执行此用例前需要满足的条件}
- **测试步骤**：
  1. {具体操作步骤}
  2. {具体操作步骤}
  3. {具体操作步骤}
- **预期结果**：{可验证的预期输出}
- **关联需求**：{对应的需求文档章节或业务规则}

---

### TC-{模块缩写}-002：{用例标题}
{同上结构}
```

## 必填字段清单

以下字段为**强制必填**，输出时不得省略任何一个：

| 字段 | 必填 | 格式要求 |
|------|:----:|----------|
| 用例 ID | **是** | `TC-{模块缩写}-{三位序号}`，如 `TC-UR-001` |
| 用例标题 | **是** | 简短描述测试目的 |
| 类型 | **是** | 必须引用 `reference/testcase-checklist.md` 中的类型名称 |
| 优先级 | **是** | `P0` / `P1` / `P2` |
| 前置条件 | **是** | 无前置条件时写"无" |
| 测试步骤 | **是** | 具体操作步骤，禁止模糊描述 |
| 预期结果 | **是** | 可客观验证的输出 |
| **关联需求** | **是** | 格式："模块名 > 子章节 > 具体规则" |

## 字段说明

### 用例 ID（必填）
格式为 `TC-{模块缩写}-{三位序号}`。模块缩写取模块名的大写首字母缩写或英文缩写：
- 用户注册 → `TC-UR-001`
- 订单管理 → `TC-OM-001`

**禁止**使用 1.1、1.2 或其他非标准编号格式。

### 类型（必填）
必须引用 `reference/testcase-checklist.md` 中定义的类型名称，保持一致。

### 优先级（必填）
- **P0**：核心正向流程，必须通过才能上线
- **P1**：重要异常流程和边界场景
- **P2**：体验优化、兼容性、非关键路径

### 测试步骤
每一步都是一个具体的用户操作或系统动作，避免"正确填写表单"这种模糊描述。应写成：
- ✗ "正确填写注册信息"
- ✓ "在手机号输入框中输入 13800138000"
- ✓ "点击获取验证码按钮"
- ✓ "在验证码输入框中输入收到的6位数字"

### 预期结果
必须是可客观验证的：
- ✗ "系统正常响应"
- ✓ "页面跳转到注册成功页，显示'注册成功'提示"
- ✓ "接口返回 HTTP 200，响应体中 status 字段值为 'success'"

### 关联需求
指向需求文档中的具体章节，格式为"模块名 > 子章节 > 具体规则"，如：
- "用户注册 > 业务规则 > 手机号格式校验"

## 模块汇总

每个文件末尾附一个汇总表：

```markdown
## 汇总

| 类型 | P0 | P1 | P2 | 合计 |
|------|----|----|-----|------|
| 正向流程 | 3 | 1 | 0 | 4 |
| 异常流程 | 0 | 5 | 2 | 7 |
| 边界值 | 0 | 2 | 1 | 3 |
| ... | | | | |
| **合计** | **3** | **8** | **3** | **14** |
```

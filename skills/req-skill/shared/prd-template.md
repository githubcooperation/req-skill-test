# 补全后需求文档模板

本文件定义 req-complete 输出的补全后需求文档的格式约定。

## 文件头

每份补全后的文档必须在最开头包含元信息块：

```markdown
---
版本: v{轮次号}.0
最后更新: {YYYY-MM-DD HH:mm}
原始文档: {原始文档路径}
审查轮次: {累计审查补全次数}
未解决项: {TBD 数量}
---
```

## 模块划分约定

文档中的每个功能模块使用 **H2 标题（`##`）** 划分。这是 req-testcase 识别模块边界的依据。

模块标题命名规则：
- 使用简短的功能名称，如"用户注册"、"订单管理"、"支付流程"
- 避免使用序号前缀（如"1. 用户注册"），直接用文字
- 模块内的子功能使用 H3 标题（`###`）

## 文档结构

```markdown
---
版本: v1.0
最后更新: 2025-03-19 14:30
原始文档: docs/prd.md
审查轮次: 1
未解决项: 2
---

# {产品/功能名称}

## 背景与目标
{简述需求的背景、解决的问题、核心目标}

## {模块A名称}

### 功能描述
{该模块做什么}

### 输入与输出
{明确的输入条件和输出结果}

### 业务规则
{逐条列出业务规则，每条规则可量化、可测试}

### 状态流转
{如果涉及状态变化，用文字或表格描述完整的状态机}

### 异常处理
{每种异常场景的处理方式}

### 约束条件
{数值范围、长度限制、格式要求、性能要求等}

## {模块B名称}
{同上结构}

## 全局约定
{跨模块的通用规则，如权限模型、数据格式、日志要求等}

## 术语表
{文档中使用的专有名词及其定义}
```

## TBD 标记

对于用户回答"不确定"或"待定"的内容，在文档中使用以下标记：

```markdown
[TBD-{ISS-ID}] {待确认的内容描述}
```

示例：
```markdown
验证码有效期为 [TBD-ISS-003] 分钟（待产品确认具体时长）。
```

这样做的好处是：
- req-testcase 的前置校验可以扫描 `[TBD-` 标记并提醒用户
- 可以通过 ISS-ID 追溯到原始缺陷
- 后续补全时容易定位

## 与原始文档的关系

补全后文档是原始文档的增强版，保留原文的所有有效内容，仅做以下修改：
- 补充缺失的信息
- 消除模糊描述（替换为具体数值或明确表述）
- 解决前后矛盾（统一为一个版本，并在 changelog 中记录）
- 按模板结构重新组织（如果原文结构混乱）

不擅自增加原文没有涉及的功能，不删除原文中明确表达的内容。

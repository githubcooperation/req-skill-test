# 缺陷条目结构定义

本文件定义 req-review 输出的缺陷清单中每条缺陷的结构。req-complete 会按此结构逐条消费。

## 字段说明

每条缺陷包含以下字段：

| 字段 | 类型 | 必填 | 说明 |
|------|------|------|------|
| `id` | string | 是 | 唯一标识，格式 `ISS-{三位序号}`，如 `ISS-001` |
| `module` | string | 是 | 所属模块名，对应文档中的 H2 标题；无法归属时填"全局" |
| `type` | enum | 是 | 缺陷类型，见下方枚举 |
| `severity` | enum | 是 | 严重程度：`P0` / `P1` / `P2` |
| `location` | string | 是 | 原文定位，格式为"章节名 > 段落关键词"或行号范围 |
| `description` | string | 是 | 缺陷描述，一句话说清楚问题是什么 |
| `context` | string | 否 | 原文相关片段的引用，帮助理解上下文 |
| `status` | enum | 是 | 处理状态：`open` / `resolved` / `tbd` |

## 缺陷类型枚举

| 类型值 | 含义 | 典型表现 |
|--------|------|----------|
| `ambiguous` | 描述模糊 | 用了"合理的"、"尽快"、"等等"这类无法量化的词 |
| `missing` | 信息遗漏 | 缺少输入输出定义、缺少异常处理说明、缺少边界条件 |
| `contradiction` | 前后矛盾 | 同一字段在不同章节描述不一致 |
| `untestable` | 不可测试 | 需求无法被转化为具体的测试步骤 |
| `incomplete_flow` | 流程不完整 | 状态流转缺少某些路径、缺少回退/取消场景 |
| `missing_constraint` | 缺少约束 | 没有说明数值范围、长度限制、格式要求等 |

## 严重程度定义

- **P0**：阻塞性问题。缺少这项信息，开发和测试都无法正常进行。典型场景：核心流程未描述、关键字段定义缺失。
- **P1**：重要问题。不影响主流程理解，但会导致实现歧义或测试遗漏。典型场景：异常流程未说明、边界条件未定义。
- **P2**：改进建议。当前描述能用但不够好，补全后能显著提升文档质量。典型场景：用词模糊但可推断、缺少示例。

## 输出格式

缺陷清单以 Markdown 表格形式输出到 `output/review-issues.md`，文件结构如下：

```markdown
# 需求文档审查报告

- **文档路径**：{原始文档路径}
- **审查时间**：{时间戳}
- **缺陷总数**：{总数}（P0: {数量} / P1: {数量} / P2: {数量}）

## 缺陷清单

| ID | 模块 | 类型 | 严重度 | 定位 | 描述 | 状态 |
|----|------|------|--------|------|------|------|
| ISS-001 | 用户注册 | missing | P0 | 注册流程 > 手机号验证 | 未说明验证码的有效期和重发间隔 | open |
| ISS-002 | 订单管理 | ambiguous | P1 | 订单状态 > 取消规则 | "合理时间内"未定义具体时限 | open |
| ... | | | | | | |

## 按模块汇总

### 用户注册
- P0: 2 条
- P1: 1 条
- P2: 0 条

### 订单管理
- P0: 0 条
- P1: 3 条
- P2: 1 条
```
